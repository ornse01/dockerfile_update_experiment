# stable と unstable の2つの Dockerfile を用意する設計方針

## 背景と問題の本質 (Context and Problem Statement)

Debian などの Linux ディストリビューションは定期的にメジャーバージョンアップが行われ、パッケージの仕様変更や非互換性などの破壊的変更が含まれることがあります。
本番環境で運用するコンテナイメージの安定性と信頼性を維持しつつ、将来の Debian メジャーバージョンアップによって生じる破壊的変更を事前に検知できる体制を整える必要がありました。
そのため、本プロジェクトではベースイメージとして `stable` と `unstable` を指定した2つの Dockerfile（`stable/Dockerfile` および `unstable/Dockerfile`）を用意し、目的に合わせた運用を行っています。

## 決定要因 (Decision Drivers)

* **本番運用の安定性**: 本番環境向けコンテナイメージにおける意図しない仕様変更や不安定性の排除。
* **破壊的変更の早期検出**: アップストリーム（Debian）の将来のメジャーバージョンアップによる影響やパッケージ依存関係の破損の早期検知。
* **リリース管理の最適化**: 本番運用しない検証用イメージの不要なレジストリ配備や管理コストの削減。

## 検討した選択肢 (Considered Options)

* **選択肢 1**: `stable` 版の Dockerfile のみを用意して運用する
* **選択肢 2**: `stable` 版と `unstable` 版の2つの Dockerfile を用意し、役割とリリース方針を分離する

## 意思決定の結末 (Decision Outcome)

選択されたオプション: **「選択肢 2: `stable` 版と `unstable` 版の2つの Dockerfile を用意し、役割とリリース方針を分離する」**

理由および各 Dockerfile の運用方針は以下の通りです。

### 1. stable 版 (`stable/Dockerfile`)
* **目的**: 本番運用を想定したコンテナイメージ。
* **ベースイメージ**: `debian:stable-slim`
* **運用・リリース方針**:
  * 高い安定性とセキュリティパッチの追従を重視。
  * CI/CD パイプラインでの検証（ビルド・脆弱性スキャン・統合テスト）を通過した後、GitHub Container Registry (GHCR) へリリース（パブリッシュ）し、デジタル署名を行って本番利用可能な状態にする。

### 2. unstable 版 (`unstable/Dockerfile`)
* **目的**: 本番運用は想定せず、将来の破壊的変更の早期検出を目的とした検証用イメージ。
* **ベースイメージ**: `debian:unstable-slim`
* **運用・リリース方針**:
  * Debian の将来のメジャーバージョンアップに伴うパッケージの変更や互換性破壊を早めに検出するため、CI/CD パイプラインでビルドおよびテストを実行する。
  * 本番運用を目的としないため、GitHub Container Registry へのリリース（パブリッシュ）は行わない。

## 影響・結果 (Consequences)

* **良い点**:
  * 本番環境の安定性を損なうことなく、次世代 Debian 環境におけるパッケージ動作やビルドの影響を日常的に監視できる。
  * 上流の破壊的変更を早期に発見・対応できるため、将来の Debian バージョン移行コストを大幅に削減できる。
  * `unstable` 版をレジストリへリリースしないことで、不要なイメージの蓄積や管理コスト・セキュリティリスクを防止できる。
* **悪い点**:
  * 2つの Dockerfile を管理する必要があるため、パッケージの追加や変更時に同期を行う手間がわずかに増える。
